# Техническое задание
# Калькулятор подбора LED‑экранов для future-screen

Документ описывает полную спецификацию разработки интерактивного калькулятора подбора LED‑экрана под мероприятие.
Цель — дать заказчику понятную рекомендацию (размер, шаг пикселя, тип установки, требования по питанию) и собрать квалифицированную заявку.

**Стек реализации:** TypeScript + Vite + React (без обязательной CMS).  
**Где размещаем:** как отдельная страница/модуль на future-screen (например, `/calculator`), с возможностью встраивания блоком на `/led` и `/support`.

---

## 1. Цели

### 1.1 Бизнес‑цели
- Увеличить конверсию в заявки (особенно из рекламы: поиск + ретаргет).
- Повысить доверие (инструмент “мы понимаем задачу”).
- Собирать заявки сразу с параметрами: формат, локация, дистанции, дата/город.
- Снизить нагрузку на менеджеров: первичная квалификация и “первый расчёт” автоматизированы.

### 1.2 Пользовательские цели
- Быстро понять, **какой LED‑экран нужен** и почему.
- Увидеть “как это будет выглядеть” (визуально и понятными цифрами).
- Получить ориентир по бюджету (вилка — опционально).
- Отправить запрос на КП/точный расчёт.

---

## 2. Принцип работы (UX‑концепция)

**Формат:** пошаговый wizard (многошаговая форма) с прогрессом и “назад/вперёд”.

**Поток:**
1) Тип мероприятия  
2) В помещении / На улице  
3) Количество зрителей  
4) Дистанция до последнего ряда (или “не знаю”)  
5) Назначение экрана (текст/видео/трансляция/фон)  
6) (опционально) Ширина сцены и ограничение по высоте  
→ Результат (рекомендация + визуализация)  
→ Форма заявки “получить КП” (с автоподставленными данными расчёта)

**Ключевой принцип:** сначала дать результат, потом просить контакт (иначе конверсия падает).

---

## 3. Входные данные (что спрашиваем у пользователя)

### 3.1 Тип мероприятия
Варианты:
- Конференция / форум
- Концерт / шоу
- Выставка / стенд
- Спорт / трансляция
- Презентация / корпоратив
- Другое

**Влияние:**
- приоритет читаемости текста (конфа/презентация)
- приоритет “эффект” и ширина задника (концерт)
- тип соотношения сторон (16:9 vs широкоформат)

### 3.2 Локация
- В помещении
- На улице

**Влияние:**
- минимальные требования к яркости и защите
- минимальный шаг пикселя (наружка обычно более крупный)
- тип установки (конструкция/ферма/опоры)

### 3.3 Количество зрителей
Ввод:
- пресеты: до 100 / 100–300 / 300–1000 / 1000+
- или ручной ввод числа

**Влияние:**
- если дистанция не задана — оцениваем дистанцию по людям (см. формулу ниже)
- оценка “размера зала” и рекомендованной высоты экрана

### 3.4 Дистанция до последнего ряда (м)
Ввод вручную (число) + чекбокс “не знаю”.

Если “не знаю”:
- рассчитываем ориентировочно на основе количества людей (см. 4.1).

### 3.5 Назначение экрана
Варианты:
- Презентации (текст важен)
- Видео / шоу
- Трансляция
- Брендинг / фон

**Влияние:**
- шаг пикселя (для текста — мельче)
- рекомендованная высота (для видео может быть ниже при той же дистанции)

### 3.6 (Опционально) Ограничения площадки
- Ширина сцены (м)
- Максимальная высота под потолком (м) / ограничение по высоте
- Способ установки: подвес / на полу / “не знаю”

**Влияние:**
- подгоняем решение под реальные ограничения, показываем “подходит/не подходит”.

## 4. Расчётная логика (MVP, реалистичная)

### 4.1 Оценка дистанции, если пользователь её не знает
Простая эвристика (на старте достаточно):
- `distance ≈ sqrt(audience) * 2`

Примеры:
- 100 → sqrt(100)=10 → ~20 м  
- 300 → ~34.6 м  
- 1000 → ~63 м

**Важно:** дать пользователю слайдер “примерно ближе/дальше” (−30% / +30%), чтобы он мог поправить оценку без измерений.

### 4.2 Рекомендованная высота экрана
Базовое правило для читабельности:
- `height = distance / 6`

Если режим “Презентации (текст)”:
- `height = distance / 5` (чуть больше)

Если режим “Брендинг/фон”:
- `height = distance / 7` (чуть меньше)

### 4.3 Рекомендованная ширина
Зависит от типа мероприятия:

- **Конференция/презентация:** 16:9  
  `width = height * 16/9 ≈ height * 1.77`

- **Концерт/шоу:** широкоформатный задник (примерно 3:1)  
  `width = height * 3`

- **Выставка/стенд:** более компактно (примерно 4:3 или 16:9)  
  по умолчанию 16:9, но ограничиваем шириной стенда (если введена)

### 4.4 Площадь
- `area = width * height`

Округление:
- height до 0.5 м
- width до 0.5 м
(чтобы получались “реальные” размеры)

### 4.5 Рекомендация шага пикселя (Pixel Pitch)
Практическое правило:
- комфортная минимальная дистанция (м) ≈ pitch(mm)

Т.е.:
- P2.6 → ~2.6 м
- P3.9 → ~3.9 м
- P5.9 → ~5.9 м
- P8 → ~8 м
- P10 → ~10 м

Логика выбора (MVP):
- distance < 3 → P2.6
- 3–6 → P3.9
- 6–10 → P5.9
- >10 → P6.6/P8/P10 (выбирать по цели и бюджету)

**Улица:** минимально P3.9 (если “очень близко” — отдельный флаг и предупреждение).

**Назначение:**
- “Презентации” → выбирать на 1 уровень “мельче” (если возможно)  
  (например, 6–10 м: вместо P5.9 может быть P3.9 при требовании текста)
- “Брендинг/фон” → можно на 1 уровень “крупнее” (если дистанция позволяет)

### 4.6 Тип установки (рекомендация)
Правила:
- Indoor + потолок высокий/подвес доступен → “подвес” (если указано)
- Indoor без подвеса → “напольная/на стойках/рама”
- Outdoor → “уличная конструкция/ферма + балласт” (и предупреждение про ветер/погоду)

### 4.7 Оценка потребления (приближённо)
Для MVP достаточно грубой вилки на м²:
- Среднее потребление: **0.2–0.4 кВт на м²**
- Пиковое: **0.6–0.9 кВт на м²**

Выводить как:
- “Ориентировочно: 6–10 кВт среднее / до 18 кВт пик”

(Формула: area * коэффициент)

### 4.8 Ориентир по бюджету (опционально)
Не показывать точную цену, только вилку:
- `budgetFrom = area * pricePerM2From`
- `budgetTo = area * pricePerM2To`

`pricePerM2From/To` задаются в конфиге (JSON), чтобы менять без кода.
Если бизнес не хочет публиковать — скрыть блок и показывать “Бюджет уточним после площадки и логистики”.

## 5. Результаты (что показываем пользователю)

### 5.1 Итоговая карточка рекомендации
Обязательные поля:
- Рекомендуемый размер: **ширина × высота (м)**
- Площадь: **м²**
- Шаг пикселя: **P…**
- Локация: indoor/outdoor (контекст)
- Тип установки: подвес/напольная/конструкция
- Питание: ориентир по кВт
- “Что входит” (минимум): доставка/монтаж/процессинг/инженер (текст из вашего оффера)

Дополнительно:
- 1–2 “пояснения почему” (коротко):
  - “Высоту подобрали по расстоянию до последнего ряда…"
  - “Шаг пикселя выбран, чтобы текст был читаем…”

### 5.2 Предупреждения/флаги
Показывать “⚠️” если:
- distance маленькая, а выбран pitch крупный → “будут видны пиксели”
- outdoor + очень близко → “нужно уточнить шаг и яркость”
- введена ширина сцены меньше расчётной → “экран будет уже, предлагаем альтернативу”

## 6. Визуализация результатов (обязательно)

### 6.1 Визуализация №1 — “Сцена в масштабе” (SVG)
Показать схему:
- прямоугольник сцены
- прямоугольник экрана (в масштабе относительно сцены)
- сектор/зона зрителей
- линия расстояния до последнего ряда

**Требования:**
- динамическое изменение размеров экрана (анимация 200–400ms)
- подписи (width/height/distance)
- адаптация под mobile (упрощённая версия)

### 6.2 Визуализация №2 — “Как будет видно” (симуляция)
Показать мини‑превью “контент на экране”:
- пример слайд‑текста (заголовок + 2 строки)
- пример видео‑кадр/логотип

Логика:
- при крупном pitch и близкой дистанции — добавляем эффект “сеточки/пикселей” и лёгкий blur
- при мелком pitch — чистая картинка

**Важно:** это не физическая симуляция, а понятная иллюстрация (честная подпись “визуальная оценка”).

### 6.3 Визуализация №3 — Сравнение (опционально, V1.5)
Кнопка “Сравнить с альтернативой”:
- показываем 2 варианта (например, P3.9 vs P5.9) с плюсами/минусами:
  - “лучше для текста / дешевле / на улице / видны пиксели”

## 7. Форма заявки и интеграции

### 7.1 Когда показываем форму
Форма показывается:
- на экране результатов (в конце)
- в виде CTA кнопки “Получить КП за 15 минут” → открывает модалку

### 7.2 Поля формы
Минимум:
- Имя
- Телефон
- Город
- Дата мероприятия (или “примерно”)
- Комментарий (опционально)

Скрытые поля (авто):
- все введённые параметры калькулятора
- все расчётные результаты
- utm_* / referrer / landing page
- версия калькулятора (v1)

### 7.3 Куда отправляем (без CRM на старте)
- Telegram (бот/чат)
- Email (копия)

**Требования:**
- антиспам: honeypot + rate limit
- лог ошибок
- подтверждение пользователю “заявка отправлена” + что дальше

## 8. Техническая реализация (TS + Vite + React)

### 8.1 Архитектура
- UI компоненты + чистые функции расчёта
- Конфиги (пичи, пороги, коэффициенты) — в JSON

Рекомендуемая структура:
```
src/
  components/
    calculator/
      Calculator.tsx
      steps/
        StepEventType.tsx
        StepLocation.tsx
        StepAudience.tsx
        StepDistance.tsx
        StepPurpose.tsx
        StepConstraints.tsx (optional)
      Result/
        ResultCard.tsx
        StageViz.tsx
        VisibilityViz.tsx
        CompareVariants.tsx (optional)
      LeadForm/
        LeadForm.tsx
  utils/
    screenMath.ts
    pitchLogic.ts
    powerEstimation.ts
  data/
    calculatorConfig.json
  pages/
    CalculatorPage.tsx
```

### 8.2 Состояние и валидация
- state: React state / Zustand (по желанию), но без тяжёлой архитектуры
- формы: react-hook-form + zod

### 8.3 Конфиг (calculatorConfig.json)
Пример структуры:
- пороги pitch
- коэффициенты высоты (1/5, 1/6, 1/7)
- коэффициенты мощности
- соотношения сторон по типу мероприятия
- вилки pricePerM2 (если показываем бюджет)

### 8.4 Производительность
- никакой тяжёлой графики; SVG предпочтительнее canvas
- memoization для расчётов
- debounce на полях ввода

## 10. Edge cases и UX‑правила

- Пользователь может пропускать “дистанцию” → калькулятор обязан выдать результат на основе эвристики.
- Везде подсказки: “если не знаете, выберите примерно”.
- Не давить на цену: лучше “ориентир” + “точно уточним после площадки”.
- Если ограничение сцены конфликтует с рекомендацией → предлагать альтернативу (уменьшить ширину, сохранить высоту или наоборот).

## 12. Критерии приёмки

- Расчёты устойчивы: не дают нулей/NaN, корректно округляют.
- На мобильных всё работает без горизонтального скролла.
- Визуализация соответствует цифрам (размеры, дистанция).
- Форма отправляет заявку и включает данные расчёта.
- Метрика фиксирует основные события.
- Время от ввода до результата — мгновенно (ощущение “считает сразу”).

## 13. Приложение: Справочные значения для pitch (для UI подсказок)
- P2.6 — близкий просмотр, презентации, конференции (минимальная комфортная дистанция ~2–3 м)
- P3.9 — универсально, сцена/зал (3–6 м)
- P5.9 — большие залы, дальше (6–10 м)
- P8 / P10 — улица, большие дистанции (10+ м)
