### 3.2 Критичные проблемы 🔴

**1. `unsafe-eval` в CSP** — [vercel.json:58-59](vercel.json#L58-L59):
```
script-src 'self' 'unsafe-inline' 'unsafe-eval' https://mc.yandex.ru
```
`unsafe-eval` существенно ослабляет защиту от XSS. Обычно нужен только для legacy‑библиотек; в текущем стеке (Vite + React) не должен требоваться. **Проверить и удалить.**

**2. `unsafe-inline` в style-src и script-src** — также снижает защиту, но менее критично; можно смягчить через nonce/hash.

**3. Rate limiting in‑memory** при Vercel Serverless не работает кросс‑инстансно: каждый инстанс имеет собственную Map, злоумышленник обходит лимит, попадая на разные холодные старты. Нужен Vercel KV / Upstash Redis / Supabase table.

**4. HSTS отсутствует** — нет заголовка `Strict-Transport-Security`. Рекомендуется:
```
Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
```

**5. tsconfig.api.json strict: false** — см. §2.1. Риск рантайм‑ошибок в обработчиках платежей/форм/Telegram.

### 3.3 Секреты
- [.env.vercel.local](.env.vercel.local) содержит `SMTP_PASS`, `SUPABASE_SERVICE_ROLE_KEY`, `TG_BOT_TOKEN`, `TG_CHAT_ID`
- ✅ Файл **присутствует в .gitignore** (строки 10–12) и не коммитится
- ⚠️ **Проверить git history** на случай исторических утечек:
  ```
  git log --all --full-history -- .env.vercel.local
  git log --all --full-history -- .env
  ```
- ⚠️ Если токен Telegram когда‑либо попадал в коммит — ротация обязательна.

### 3.4 Аутентификация
- Flow: Supabase `signInWithPassword` → сессия в localStorage (Supabase SDK по умолчанию)
- `onAuthStateChange` слушается в [src/context/AuthContext.tsx](src/context/AuthContext.tsx)
- ⚠️ Сессия хранится в localStorage — уязвимо к XSS. httpOnly cookies (через `@supabase/auth-helpers`) предпочтительнее, но требуют SSR‑адаптера.

### 3.5 Инъекции / XSS
- `dangerouslySetInnerHTML` использовать только после `DOMPurify.sanitize`
- Проверить, что **все** места, где HTML рендерится от пользователя/БД, проходят через [src/lib/sanitize.ts](src/lib/sanitize.ts)
